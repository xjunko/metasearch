JSON API

query the web, images, and news over a single authenticated endpoint and get back clean, structured JSON: the exact same data that powers the results pages, with none of the HTML.

the API is off by default. the instance author enables it by adding a key to the database, then shares that key with whoever should have access. there are no rate limits or pagination caps: you get whatever the upstream returns.

enabling it (instance authors)

keys live in a D1 database, one row per key, so you can add, label, and revoke them individually. create the database once and apply the migration: 

wrangler d1 create metasearch
# paste the printed database_id into wrangler.toml, then:
wrangler d1 migrations apply metasearch --remote

each key is a random string with an optional label. mint one and hand it out:

KEY=$(openssl rand -hex 32)
wrangler d1 execute metasearch --remote \
  --command "INSERT INTO api_keys (key, label) VALUES ('$KEY', 'alice')"
echo "$KEY"

list who has access, and revoke a single key without touching the others:

# list
wrangler d1 execute metasearch --remote \
  --command "SELECT label, created_at FROM api_keys"

# revoke just one
wrangler d1 execute metasearch --remote \
  --command "DELETE FROM api_keys WHERE label = 'alice'"

auth is a single read, no per-request writes. with no database bound, the endpoint stays disabled and answers 503.

making a request

send a POST to /api with a Bearer token and a JSON body. 

POST /api
Authorization: Bearer YOUR_KEY
Content-Type: application/json

{
  "query": "metasearch",
  "type": "web",
  "page": 0
}

request fields

field type description
query string required. the search query.
type string optional, defaults to web. one of web, images, news.
page integer optional, defaults to 0. the result offset. 0 is the first page; increment to paginate. no upper bound.

examples

curl

curl -X POST https://search.kafu.ovh/api \
  -H "Authorization: Bearer YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"query":"metasearch","type":"web","page":0}'

javascript

const res = await fetch("https://search.kafu.ovh/api", {
  method: "POST",
  headers: {
    Authorization: "Bearer YOUR_KEY",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ query: "metasearch", type: "web", page: 0 }),
});

const data = await res.json();
console.log(data.results);

responses

every successful response echoes query, type, and page, exposes more_results_available so you know whether to fetch the next page, and carries the results for the requested type.

web (type: "web")

results is an object grouping the different result kinds. fields are null when absent for a given query. 

{
  "query": "metasearch",
  "type": "web",
  "page": 0,
  "badResults": false,
  "more_results_available": true,
  "results": {
    "web": {
      "results": [
        {
          "title": "…",
          "url": "https://…",
          "description": "…",
          "age": "…",
          "meta_url": { "hostname": "…", "favicon": "…" },
          "profile": { "name": "…", "img": "…" },
          "thumbnail": { "src": "…" },
          "deep_results": { "buttons": [{ "title": "…", "url": "…" }] },
          "cluster": [{ "title": "…", "label": "…", "url": "…", "description": "…" }]
        }
      ]
    },
    "news": { "results": [ … ] },
    "videos": { "results": [ … ] },
    "discussions": { "results": [ … ] },
    "faq": { "results": [ … ] },
    "infobox": { "results": [ … ] },
    "rich": [ … ],
    "qanda": …,
    "locations": …,
    "images": …,
    "mixed": [ … ]
  }
}

rich carries instant answers (calculator, weather, currency, crypto, timezones, unit conversions, and more), each tagged with a subtype. mixed mirrors the upstream ordering of the main column.

pagination

start at page: 0. if the response has more_results_available: true, request page: 1, then 2, and so on. when it is false there is nothing more to fetch.

errors

errors come back as JSON with an error string and a matching status code.

status meaning
400 bad body, missing query, unknown type, or invalid page.
401 missing, malformed, or invalid Authorization header.
502 the upstream search failed.
503 the API is not enabled on this instance. 
{ "error": "missing required field: query" }

notes

  • CORS is open, so the endpoint is reachable from the browser. only do that if you accept exposing your key client-side.
  • no rate limits are enforced by the API itself. be a good citizen: the upstream is shared.
  • no logging. queries sent through the API are not stored, same as the website.
